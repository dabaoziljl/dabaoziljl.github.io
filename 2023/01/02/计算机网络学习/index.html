第一章 计算机网络概述

物理层做数字信号和物理信号的转换，数据链路层提供点到点（一跳一跳）的服务，网络层基于点到点的服务建立端到端的服务，路由/sdn，传输层在网络层基础上加强细分服务。

1.1 什么是Internet？

互联网：网络的网络

互联网的物理构成

  • 节点：主机节点，数据交换节点（交换机，路由器）。路由器工作在网络层，交换机工作在链路层。
  • 边：接入网链路（主机连接到互联网的链路），主干链路（路由器间的链路）。
  • 协议：对等的实体在通信的过程中需要遵守的标准,包括语法，语义和动作。

如何提出协议？

IETF：Internet Engineering Task Force

RFC: Request for comments

互联网的构成，从服务的角度

分布式应用进程，以及为分布式应用进程提供通信服务的基础设施。

1.2 网络边缘

  • C/S模式：有专门的服务器，等候客户端的主动请求。可扩展性问题，服务器可能会忙不过来或宕机。
  • P2P模式：几乎没有一直运行的服务器；

面向连接和有连接的区别

面向连接指，连接状态只在端系统上维护，而中间设备均不知道；而有连接则是端系统和中间的连接设备均知道这个连接。

1.3 网络核心

  • 电路交换：常用在传统电话网中。独享资源，不共享，性能有保障，但是容易产生资源的浪费。频分，时分，波分，将带宽分成片。不适合计算机间的通信，因为连接建立时间长；计算机间的通信具有突发性，易造成浪费；可靠性不高，核心节点影响大范围通信。
  • 分组交换：将信息打成分组，节点间链路不再细分，分组在每个节点中存储转发。资源共享，按需使用，像是更高级的时分复用（统计多路复用）。会有排队延迟和丢失。
    • 数据报。每个分组携带完整的源和目标的信息，源和目标主机不需要握手，无连接，每个分组单独转发。
    • 虚电路。源和目标主机握手，在交换节点之间保持通信状态，建立虚拟的电路，每个分组走相同的路径。

1.4 接入网和物理媒体

  • modem 调制解调器，使用电话线
  • DSL
  • 线缆网络

1.5 Internet结构和ISP

​ 端系统通过接入ISP接入到互联网，接入ISP相应的必须是互联的。acces ISP， regional ISP，global ISP，ISP间的合作， IXP，以及一些ICP。

1.6 分组延时，丢失和吞吐量

四种分组延时

  • 处理延时：查路由表，检查差错等操作，很少；
  • 排队延时：在输出链路上等待传输的时间，取决于拥塞程度;流量强度$L=\frac{La}{R}$,流量强度为1时平均排队时延无穷大；
  • 传输延时：分组打出去的时间，分组长度*传输速度；
  • 传播延时：信息在物理链路上消耗的时间；

丢包，可能由上一级节点重传，可能由源主机重传，可能根本不重传。

吞吐量，源端和目标端之间传输的速率（数据量/单位时间）。

1.7 协议层次和服务模型

​ 将网络复杂的功能分成功能明确的层次。本层协议实体相互交互执行本层的协议动作，目的是实现本层功能，通过接口为上层提供更好的服务。

Internet协议栈

  • 物理层：将数字数据转换为物理信号，并传输；
  • 链路层：在相邻节点间，传输以帧为单位的数据；
  • 网络层：传输以分组为单位的端到端的数据；
  • 传输层：实现进程到进程的区分，将网络层可能的不可靠服务变为可靠的服务；
  • 应用层：完成应用报文 之间的交互；

ISO/OSI参考模型

  • 会话层：数据交换的同步，会话管理；
  • 表示层：表示转换；

第二章 应用层

2.1 应用层协议原理

  • 进程通信：不同主机，通过交换报文来实现通信
    • 服务器进程：等待连接的进程
    • 客户端进程：主动发起通信的进程
  • 进程通信需要解决的问题
    • 进程的标识和寻址
      • IP+TCP/UDP+端口
    • 应用层如何使用传输层的服务（形式，地点）
      • 给定发送者，接受者以及交付的货物，进行报文的传输
      • Socket：在本机， 用整数代表TCP四元组（源IP+端口，目标IP+端口）/UDP二元组（本地IP+端口），方便管理；代表了会话关系
    • 应用报文的定义，格式和解释（应用层协议 ）
  • 安全TCP—SSL
    • 实现在应用层，提供加密的TCP连接，私密性，数据完整性等。
    • https = http + ssl + tcp

2.2 Web和HTTP

  • Web页：由一些对象组成，包括html文件，图像，小程序等。Web页包含一个基本的html文件，以及若干对象的引用URL。

  • HTTP：超文本传输协议，无状态协议；

  • cookies：将http变成有状态的。

    • 客户端在第一次发送请求时，不带cookie；服务器返回时新建一个cookie并返回给客户端；下一次访问同一个服务器时，在请求中会自带cookie，便于服务器分辨信息；

  • HTTP请求报文：

    • 请求行：GET/POST/HEAD 命令
    • 首部行：包含一些重要信息
    • 实体行

  • HTTP相应报文：

    • 状态行： 协议，版本号，状态码
    • 首部行
    • 实体行

  • Web缓存

    • 不访问原始服务器，就满足客户的请求；
    • 条件GET：在报文头部添加 modified since字段，在缓存中查找之前先往目标服务器查询，如果给请求目标没有修改，则使用缓存中的内容，否则从目标服务器获取并更新缓存。

  • HTTP1.0

    • 非持久连接，一次连接并完成数据传输后，连接中断；
    • 响应时间：一个RTT用来发起TCP连接，一个RTT用来HTTP请求并等待HTTP相应；2*RTT+传输时间；
    • GET/POST/HEAD

  • HTTP1.1

    • 持久连接，一次连接并完成数据传输后，连接不中断；
      • 非流水线：发出一个对象请求，等待请求回来后再发出下一个对象请求；
      • 流水线（HTTP1.1默认形式）：发出一个对象请求，不等待请求返回就发出下一个对象请求；
    • GET/POST/HEAD/PUT/DELETE

2.3 FTP

  • 从远程主机上传输文件或从远程主机上接受文件。
  • 控制连接（带外）和数据连接分离。
  • 有状态协议

2.4 EMail

三个主要组成部分

  • 用户代理；
  • 邮件服务器；
  • 发送(SMTP)和拉取(POP3，IMAP，HTTP)邮件；

HTTP与SMTP的区别

每个HTTP报文中仅包含一个对象，但每个SMTP报文中可以有多个对象；

  • SMTP
    • 用户端向自己的邮件服务器发送邮件
    • 自己的邮件服务器向对方的邮件服务器发送邮件
    • 报文内容全为 7位ASCII码
    • 使用持久连接
    • 多媒体邮件扩展MIME，或用base64编码以满足SMTP的要求
  • POP3
    • 无状态
    • 本地管理文件夹
  • IMAP
    • 有状态
    • 允许用户远程管理文件夹，允许用目录来组织报文

2.5 DNS

​ 主要负责IP地址和主机域名之间的转换。还有主机别名到规范名字的转换，邮件服务器的别名和正规名字的转换， 负载均衡。

设备上网必备信息

​ 主机IP地址，子网掩码，网关，local name server

Local name server如何工作

​ 若本地有缓存，则直接返回；若没有缓存，递归查询/迭代查询（踢皮球）；

  • 分层的
    • 层次树状结构的命名方法；顶级域，通用的，国家的，
  • 分布式
    • 13个根名字服务器
    • 区域内的权威服务器
    • 使用资源记录RR，记录对应格式信息
  • 运行在UDP的53号端口；
  • 是Internet的核心功能，但是在应用层实现；

2.6 P2P应用

​ 没有(或极少)一直运行的服务器，任意端系统都可以直接通信。

  • 非结构化 P2P
    • peer与peer之间的连接关系可以是任意的
    • 集中化目录：每个节点上线时，需要像集中服务器注册和登记（我的IP和我有的资源 ）；
    • 完全分布式：泛洪，向所有邻居发出查询；
    • 混合体：每个实体要么是组长，要么隶属于一个组长，组长跟踪组员的内容；
    • 优先向对我好的节点提供信息，并每隔一定时间随机选择节点提供信息；稀缺优先请求；
    • 如何加入洪流？带外解决。torrent文件给出tracking server，并返回peer列表。
  • 结构化P2P
    • peer与peer之间的连接关系是环状，树状等有序拓扑结构
    • 每一个节点对应一个hash值，每一个节点有一定hash值范围对应的内容块；

2.7 CDN

​ 多媒体流化服务：DASH – Dynamic Advanced Streaming over HTTP。根据告示文件，自适应地从对应的服务器下载我需要的视频清晰度对应的数据块。

​ ICP向CDN购买服务。CDN在全球网络中部署网络缓存节点，预先将一些内容存放在缓存节点中，便于客户端快速的访问内容。客户端需要获取文件时，首先向源服务器获取告示文件，然后根据需求从告示文件中说明的IP处获取文件。

部署CDN缓存节点的策略

  • enter deep：将缓存部署在local ISP中。能够离客户端更近，提供的服务质量更高，但是需要部署很多的节点。
  • bring home：仅将缓存部署在一些关键的位置。

2.8 TCP套接字编程

流程

  • 服务器先运行，创建欢迎socket，并与本地端口捆绑，阻塞式地等待接受用户的连接请求；
  • 客户端创建本地socket ，与本地端口捆绑，并与服务器请求连接；
  • 当客户端请求到来时，同意连接建立，创建新的连接socket与客户端通信；
  • 在连接socket上成功建立TCP连接，欢迎socket继续守候；

第三章 传输层

3.1 概述和传输层服务

​ 传输层提供的服务是，进程到进程之间的，以message为单位的逻辑通信。

​ 发送方传输层会将应用层的报文分成报文段并递给网络层，接收方将收到的报文段重组并传递给应用层。

复用解复用

多个应用进程用同一个传输层服务进行报文发送，多个报文被看作一个大报文，称为复用；接受方的传输层将收到的报文分解并分别送给对应的应用进程，称为解复用。

3.2 多路复用与解复用

​ 靠端口号实现复用和解复用。 从多个套接字接受来自多个进程的报文，根据套接字对应的IP地址和端口号对报文段用头部加以封装；在解复用时，利用报文段中的头部信息将收到的报文段分给对应的套接字。

​ 在UDP中，使用目标IP和目标端口号对应socket，找到目标应用；在TCP中，使用四元组唯一确定socket，找到目标应用。

3.3 无连接传输 UDP

​ 在网络层IP协议的基础上，仅额外提供了多路复用解复用的服务。

​ 使用UDP通信时，无需握手，多用于流媒体，DNS等。

UDP校验和。如果出错，UDP数据包会被丢失。

发送时通过某种规则计算校验和并放在字段中，接受时再计算一次进行比较。不相等则说明有错，相等不一定说明不错。

3.4 可靠数据传输RDT的原理

如何基于不可靠的信道实现可靠数据传输。

RDT1.0 在可靠信道上的可靠数据传输

下层信道完全可靠，仅需封装解封装信息后发送即可。

RDT2.0 具有比特差错的信道

采用差错控制编码进行差错检测。

发送方发送信息后，等待接受方的ACK/NAK。接受方收到信息后进行差错校验，并返回ACK/NAK。发送方收到NAK则重传，ACK则继续发送新的信息。

缺陷：ACK/NAK出错。

RDT2.1发送方处理出错的ACK/NAK

停止等待协议。发送方给发送的信息编号，如果ACK/NAK出错，发送方直接再发一次，等待接受方的ACK。接受方如果第二次收到相同编号的信息，则再次发送ACK。

RDT2.2 无NAK的协议

对当前分组的反向确认，用对之前分组的正向确认来代替。

比如说，接受第一个分组出错了，返回ACK0而不是返回NAK1。

RDT3.0 具有比特差错和分组丢失的信道

发送端超时重传：如果到时没有收到ACK，则重传；如果只是延误了，通过序列号可以解决这个问题。如果是超时设置过短，也可以通过序号解决重复问题。

与RDT2.2不同的点在于，发送方发送P1接受到ACK0时，会等待超时重传P1而不是立即发送P1。

缺陷：信道利用率极低，信道中同时只有一个信息在传递。 网络协议限制了物理资源的利用。

流水线协议

允许发送方在未得到对方确认的情况下一次发送多个分组。

  • 滑动窗口协议
    • 发送缓冲区和发送窗口SW
      • 用于存放已发送但没有得到确认的分组
      • 前沿随着分组发送不断前移，直到达到发送缓冲区边界；后沿随着老分组的确认不断前移；
    • 接受缓冲区和接受窗口RW
      • 控制哪些分组可以接受
      • RW=1则只能顺序接受；RW>1则可以乱序接受
  • 回退N步协议GBN
    • SW>1, RW=1
    • 累计确认，发送连续收到的最大的分组确认，仅对第一个未确认的分组有超时定时器
    • 窗口最大尺寸 $2^n - 1$
  • 选择性重发协议SR
    • SW>1, RW>1
    • 非累计确认，收到什么分组就发送那个分组的确认，每个未确认的分组都有一个超时定时器
    • 窗口最大尺寸$2^{n-1}$

3.5 面向连接的传输 TCP

  • 点对点
  • 可靠的，有序的字节流
  • 流水线，有发送缓存和接受缓存
  • 全双工数据，在同一连接中数据流是双向的
  • 面向连接
  • 有流量控制

TCP报文结构

  • 序号与确认号

    • 序号为报文段首字节在字节流中的编号
    • 确认号，期望从另一方收到的下一个字节的序号

  • 往返延时RTT和超时

    • $EstimatedRTT = (1-\alpha)*EstimtedRTT + \alpha * SampleRTT$
    • $DevRTT = (1-\beta)*DevRTT + \beta *|SampleRTT-EstimatedRTT|$
    • 超时时间=$EstimatedRTT + 4 * DevRTT$

可靠数据传输

GBN和SR的合并体。仅有单个重传定时器，给最老的那个段定时；累计确认，接受方返回顺序到来的最后一个字节+1的ACK；若超时，仅重发最老的那个段，选择重传；若收到重复（3个）的确认，则快速重传；接受方收到合理的段，会延迟确认，等待下一个报文段，若收到下一个报文段则立即ACK，若等待时间内没收到则ACK当前字节；对于乱序到达的段，TCP未规定是保留还是丢弃，立即返回原本的ACK字节。

捎带技术piggybacking

正常的全双工交流中，双方各自数据发数据，确认发确认。捎带中，确认以及其他信息会附加在数据中以减少流量。

流量控制

接受方控制发送方，不让发送方发送太快以至于接受方缓冲溢出。

接受方在向发送方的报文段头部rwnd字段告知空闲缓存大小。

连接管理

  • 两次握手是否总是可行
    • 连接请求超时重发，导致在服务器端建立了半连接
    • 连接请求超时重发以及数据超时重发，导致老的数据被当成新的数据接受了
  • 三次握手
    • 发送端告知自己的初始序号；
    • 接受端确认发送端的序号并告知自己的序号；
    • 发送端确认接受端的序号；
  • 四次挥手
    • 两个方向的TCP连接分别拆除，对称释放
    • 并不完美
